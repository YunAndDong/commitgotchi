## 0. Document Purpose

이 PRD는 commitgotchi MVP를 함께 만드는 PM·백엔드(Spring Boot)·AI(FastAPI)·프런트엔드 담당과 다운스트림 워크플로(아키텍처·UX·에픽/스토리) 소유자를 위한 문서다. Product Brief(`brief-commitgotchi-2026-06-07`)를 토대로 작성했으며 브리프를 중복하지 않고 그 위에 요구사항을 쌓는다. 구조는 Glossary로 어휘를 고정하고, 기능(§4)을 그룹화해 각 기능 아래에 전역 번호의 기능 요구사항(FR)을 중첩한다. 추론한 부분은 본문에 `[ASSUMPTION]`으로 인라인 표기하고 §9에 색인했다. 시스템 구성·메시지 스키마·내부 API 계약 등 기술 상세는 같은 폴더의 `addendum.md`에 보존돼 아키텍처 단계로 넘어간다.

## 1. Vision

commitgotchi는 혼자 CS를 공부하는 사람의 매일의 학습을 가상 캐릭터의 성장으로 바꿔주는 학습 동반자다. 사용자가 하루의 학습을 리포트로 남기고 추천 퀴즈를 풀면, AI가 이를 분석·채점해 피드백과 다음 학습 방향을 돌려주고, 그 결과가 캐릭터의 다섯 능력치와 감정·진화 상태로 반영된다. 공부한 만큼 내 분신이 자라고 진화하기 때문에 "오늘도 공부할 이유"가 생긴다.

혼자 하는 CS 학습은 피드백이 없고 진척이 눈에 보이지 않아 쉽게 무너진다. commitgotchi는 AI 기반의 객관적 평가와 게임화된 성장 루프를 결합해 이 문제를 정면으로 다룬다. 이번 포트폴리오 MVP가 가장 증명하려는 가치는 **AI 일일 레포트·채점·피드백·추천의 품질** — 즉 AI가 내놓는 평가와 추천이 실제로 믿을 만하고 학습에 도움이 되는가다.

성공하면 CS를 넘어 어학·자격증 등 다양한 학습 도메인으로 캐릭터 성장 모델을 확장하고, 공유·랭킹을 키워 "혼자 하지만 함께 자라는" 학습 커뮤니티로 발전한다. [ASSUMPTION: 장기 비전, 브리프에서 계승]

## 2. Target User

### 2.1 Jobs To Be Done

- **취준·이직 준비 개발자 (1차 핵심):** "CS 면접·코딩테스트를 혼자서도 피드백 받으며 매일 꾸준히 준비하고 싶다."
- **CS 전공 학생:** "DB·알고리즘·네트워크 등 전공 과목을 공부하며 내 약점을 객관적으로 점검하고 싶다."
- **비전공 입문자:** "개발 입문 단계에서 무엇을 어떤 순서로 공부할지 길잡이와 학습 습관을 갖고 싶다."
- **공통 정서적 JTBD:** "내 노력이 눈에 보이는 성장으로 쌓이는 감각을 느끼며 중도 이탈하지 않고 싶다."

### 2.2 Non-Users (v1)

- 팀·스터디 단위의 협업 학습 관리를 원하는 그룹 (v1은 개인 단독 학습에 집중).
- CS 외 도메인(어학·자격증 등) 학습자 — 콘텐츠/추천이 CS에 특화돼 있어 v1 범위 밖.
- 모바일 네이티브 앱만 쓰려는 사용자 — v1은 웹 우선.

### 2.3 Key User Journeys

- **UJ-1. 윤석, 퇴근 후 오늘 공부를 캐릭터에게 먹인다.**
  - **Persona + context:** 윤석, 이직 준비 중인 3년차 백엔드 개발자. 매일 1~2시간 CS를 혼자 공부하지만 진척이 안 느껴져 동기가 식어간다.
  - **Entry state:** 이미 로그인된 웹 세션. 활성 캐릭터 "커밋 몬스터"가 대시보드에 떠 있다.
  - **Path:** 대시보드에서 오늘의 학습 리포트 작성 → "Spring JPA N+1 문제와 해결법"을 제목·본문으로 저장 → 어제 AI가 추천한 퀴즈 3문항을 풀고 제출.
  - **Climax:** 다음날 오전 9시 전까지 AI 일일 레포트가 도착 — 채점 결과·피드백·점수 변화량(알고리즘 +3, 네트워크 +1)이 캐릭터에 반영돼 전투력이 오르고, 캐릭터 감정이 "기쁨"으로 바뀌며 칭찬 메시지를 띄운다.
  - **Resolution:** 윤석은 다음 학습 추천 주제를 보고 "내일은 이걸 하면 되는구나" 하고 닫는다.
  - **Edge case:** AI 처리가 실패하면 기본 처리로 점수가 0 변화/기본 메시지로 표시되고, 흐름은 끊기지 않는다.

- **UJ-2. 지민, 첫 캐릭터를 만든다.**
  - **Persona + context:** 지민, 비전공 입문자. 무엇부터 공부할지 막막하다.
  - **Entry state:** 방금 가입·로그인 완료, 보유 캐릭터 0개.
  - **Path:** 캐릭터 생성 → 이름과 키워드·성격("칭찬 많지만 틀린 건 명확히 짚는") 입력 → AI가 캐릭터 이미지를 생성 → 자동으로 활성 캐릭터로 지정.
  - **Climax:** 내 분신이 화면에 등장하고 능력치 5종이 0에서 시작하는 걸 본다.
  - **Resolution:** 첫 학습 리포트를 쓸 준비가 된다.
  - **Edge case:** AI 이미지 생성 실패 시 기본 이미지 세트 중 하나가 배정되고 생성은 정상 완료된다.

- **UJ-3. 윤석, 랭킹과 공유 게시판을 둘러본다.**
  - **Persona + context:** UJ-1의 윤석, 며칠 학습이 쌓인 뒤.
  - **Entry state:** 로그인 상태, 활성 캐릭터 전투력 누적.
  - **Path:** 랭킹에서 전투력 기준 내 순위 확인 → 공유 게시판에서 다른 사용자가 공유한 캐릭터 열람 → 마음에 드는 캐릭터에 리뷰 작성.
  - **Climax:** 자기 캐릭터를 게시판에 공유해 다른 사람의 리뷰를 받는다.
  - **Resolution:** 경쟁·비교가 다음 학습 동기로 이어진다.

**Scope dial:** UJ-1은 핵심 루프이므로 heavier(흐름·엣지 포함). UJ-2/UJ-3은 보조.

## 3. Glossary

*다운스트림 워크플로와 독자는 아래 용어를 그대로 사용한다. 동의어 금지.*

- **사용자(User)** — 가입·로그인한 학습자. 캐릭터를 최대 3개 보유.
- **Role** — 사용자의 접근 권한 구분. `USER`와 `ADMIN` 두 종류이며 공개 회원가입의 기본값은 `USER`.
- **Access Token** — 보호 API 접근에 사용하는 짧은 유효기간의 서명된 JWT.
- **Refresh Token** — Access Token 재발급에 사용하는 장기 토큰. 원문이 아니라 해시값으로 저장하고 Rotation 및 폐기 상태를 관리한다.
- **보호 API(Protected API)** — 유효한 Access Token이 있어야 접근할 수 있는 API.
- **관리자 API(Admin API)** — 인증된 사용자 중 `ADMIN` Role만 접근할 수 있는 API.
- **캐릭터(Character)** — 사용자의 학습을 성장으로 표현하는 가상 분신. 능력치·감정·진화 상태·이미지를 가진다. 사용자당 최대 3개.
- **활성 캐릭터(Active Character)** — 사용자가 현재 활성화한 단 하나의 캐릭터. 학습 점수는 활성 캐릭터에만 반영된다. 사용자당 항상 정확히 1개(보유 캐릭터가 있을 때).
- **능력치(Stat)** — 캐릭터의 학습 역량 수치. 5종: DB, 알고리즘, CS, 네트워크, 프레임워크.
- **전투력(Battle Power)** — 능력치 5종의 총합. 랭킹·진화 판정 기준.
- **진화(Evolution)** — 캐릭터가 기본 단계에서 진화 단계로 1회 변하는 사건. 전투력 1,000점 이상 도달 시 발생. 캐릭터당 최대 1회.
- **감정(Emotion)** — 캐릭터의 현재 상태 표현. 기쁨·슬픔·화남 중 하나.
- **상태 메시지(Status Message)** — 캐릭터의 감정·학습 결과를 반영해 표시되는 한 줄 메시지.
- **학습 리포트(Study Log)** — 사용자가 하루 동안 작성·저장하는 그날의 학습 기록(제목·본문).
- **퀴즈(Quiz)** — 학습 리포트·답안 기반으로 AI가 추천·생성한 문제. 모범답안을 가진다.
- **모범답안(Model Answer)** — 퀴즈에 대해 AI가 보유한 기준 답안. 채점 근거.
- **퀴즈 제출(Quiz Submission)** — 사용자가 퀴즈에 제출한 답안.
- **채점(Grading)** — AI가 퀴즈 제출 답안을 모범답안 대비 평가하는 과정. **제출 즉시 동기 수행**(흐름 B).
- **피드백(Feedback)** — 채점 결과에 따라 AI가 생성하는 강·약점 및 개선 안내.
- **다음 학습 추천(Next Study Recommendation)** — 그날 학습 분석으로 AI가 제시하는 다음날 학습 주제.
- **AI 일일 레포트(AI Daily Report)** — AI가 하루 **학습 리포트**를 분석해 자정 배치로 생성하는 결과물. 분석·점수 변화량(학습분)·그날 퀴즈 결과 종합 코멘트·다음 학습 추천·추천 퀴즈를 포함. 퀴즈 채점 자체는 즉시 채점(흐름 B)에서 별도 수행.
- **점수 변화량(Score Delta)** — AI 일일 레포트가 산출한 능력치별 증감. 활성 캐릭터에 일 단위로 누적된다.
- **주간 점수 변화량(Weekly Score Delta)** — 최근 7일 점수 변화량 집계. 표시·통계·AI 분석 컨텍스트 용도이며 점수 반영의 단위가 아니다.
- **주간 학습 스트릭(Weekly Study Streak)** — 최근 7일 학습 여부를 나타내는 7자리 표식(예: `0100011`).
- **랭킹(Ranking)** — 활성 캐릭터 전투력 기준 사용자 순위.
- **대시보드(Dashboard)** — 활성 캐릭터·능력치·전투력·최근 학습 현황을 보는 사용자 홈.
- **공유 게시글(Shared Post)** — 사용자가 캐릭터를 공유 게시판에 올린 글.
- **리뷰(Review)** — 공유 게시글에 다른 사용자가 다는 평가/코멘트.
- **Fallback** — AI 처리(이미지 생성·채점·레포트 생성) 실패 시 흐름을 끊지 않기 위한 기본 처리.

## 4. Features

### 4.1 회원 관리 (Account)

**Description:** 사용자는 이메일·비밀번호로 가입하고 로그인하며, 서명된 JWT Access Token과 Rotation되는 Refresh Token으로 보호 API를 이용한다. 일반 보호 API는 `USER`와 `ADMIN`이 접근할 수 있고 관리자 API는 `ADMIN`만 접근할 수 있다. 실현 UJ-2.

**Functional Requirements:**

#### FR-1: 회원 가입
사용자는 이메일·비밀번호로 계정을 생성할 수 있다.

**Consequences (testable):**
- 이메일은 사용자 간 중복될 수 없으며 중복 이메일 가입 시 `409 Conflict`가 반환된다.
- 비밀번호는 BCrypt 등 안전한 단방향 해시 방식으로 저장되고 평문은 저장되지 않는다.
- 공개 회원가입으로 생성되는 계정의 Role은 항상 `USER`다.
- 공개 회원가입 요청은 Role을 선택하거나 `ADMIN` 계정을 생성할 수 없다. 요청에 Role 필드가 포함되어도 `ADMIN` 계정이 생성되지 않아야 한다.
- 이메일 형식, 비밀번호 정책 등 입력값 검증 실패 시 `400 Bad Request`와 필드 또는 원인을 식별 가능한 오류가 반환된다.

#### FR-2: 로그인 / JWT Access Token 발급·검증
사용자는 올바른 자격 증명으로 로그인해 서명된 Access Token과 Refresh Token을 함께 발급받고, 유효한 Access Token으로 보호 API에 접근할 수 있다.

**Consequences (testable):**
- 잘못된 비밀번호 또는 존재하지 않는 이메일로 로그인하면 `401 Unauthorized`가 반환되며, 응답은 계정 존재 여부를 추가로 노출하지 않는다.
- Access Token에는 최소한 사용자 식별자, Role, 발급 시각, 만료 시각이 포함된다.
- Access Token은 짧은 유효기간을 사용한다. `[ASSUMPTION: 정확한 유효기간은 아키텍처 단계에서 확정]`
- 서버는 모든 보호 API 요청에서 JWT 서명, 형식, 만료 여부를 검증한다.
- 누락·변조·만료·잘못된 형식의 Access Token으로 보호 API에 접근하면 `401 Unauthorized`가 반환된다.
- JWT 비밀키는 소스코드나 Git 저장소에 포함하지 않고 환경변수 또는 외부 Secret 관리 수단으로 주입한다.
- 운영 환경은 충분히 긴 JWT 비밀키를 사용해야 한다. `[ASSUMPTION: 최소 길이 및 서명 알고리즘은 아키텍처 단계에서 확정]`

#### FR-24: Refresh Token 저장·재발급·Rotation
시스템은 로그인 성공 시 Refresh Token을 발급하고, 유효한 Refresh Token 제출 시 새 Access Token과 새 Refresh Token을 발급한다.

**Consequences (testable):**
- Refresh Token 원문은 DB에 저장되지 않고 안전한 해시값으로 저장된다.
- 저장된 Refresh Token 레코드는 사용자, 만료 시각, 폐기 여부 또는 폐기 시각과 연결된다.
- 재발급 시 Refresh Token Rotation을 적용해 기존 Refresh Token을 폐기하고 새 Refresh Token을 발급한다.
- 만료·폐기·변조되었거나 존재하지 않는 Refresh Token으로 재발급할 수 없다.
- Rotation 또는 로그아웃으로 폐기된 Refresh Token이 재사용되면 인증 실패로 처리된다.

#### FR-25: 로그아웃
로그인된 사용자는 Refresh Token을 제출해 해당 로그인 세션을 로그아웃할 수 있다.

**Consequences (testable):**
- 로그아웃 시 제출된 Refresh Token이 폐기되며 이후 재발급에 사용할 수 없다.
- Access Token은 무상태 JWT이므로 로그아웃 후에도 만료 전까지 기술적으로 유효할 수 있다.
- 로그아웃 이후 Access Token 잔여 위험을 완화하기 위해 Access Token 유효기간을 짧게 유지한다.

#### FR-26: Role 기반 인가
시스템은 사용자 Role을 `USER`와 `ADMIN`으로 구분하고, 인증된 요청의 Role에 따라 API 접근을 허용하거나 거부한다.

**Consequences (testable):**
- `USER`와 `ADMIN`은 모두 일반 보호 API에 접근할 수 있다.
- 관리자 전용 API에는 `ADMIN`만 접근할 수 있다.
- 인증되지 않은 요청은 `401 Unauthorized`, 인증되었으나 Role 권한이 부족한 요청은 `403 Forbidden`으로 구분한다.
- Role을 Access Token Claim에 포함하는 경우 서버는 Claim을 신뢰하기 전에 JWT 서명을 반드시 검증한다.

#### FR-27: 최소 인증·인가 검증 API
시스템은 인증·인가 구현을 종단 검증할 수 있는 최소 보호 API와 관리자 API를 제공한다.

**Consequences (testable):**
- `GET /api/users/me`는 인증된 `USER`, `ADMIN`이 접근할 수 있고 현재 인증된 사용자의 식별 정보와 Role을 반환한다.
- `GET /api/admin/ping`은 `ADMIN`만 접근할 수 있다.
- `USER`가 `GET /api/admin/ping`에 접근하면 `403 Forbidden`이 반환된다.

#### FR-28: 인증·인가 실패 응답
시스템은 인증·인가 및 회원가입 실패를 일관되고 민감정보를 노출하지 않는 오류 응답으로 제공한다.

**Consequences (testable):**
- 잘못된 로그인 정보와 토큰 누락·변조·만료·잘못된 형식은 `401 Unauthorized`로 응답한다.
- 인증은 되었지만 Role 권한이 부족하면 `403 Forbidden`으로 응답한다.
- 이메일 중복은 `409 Conflict`, 입력값 검증 실패는 `400 Bad Request`로 응답한다.
- 오류 응답은 최소한 HTTP 상태 코드, 애플리케이션 오류 코드, 사용자에게 노출 가능한 오류 메시지, 요청 추적 정보 또는 발생 시각 정보를 일관된 형식으로 제공한다.
- 오류 응답과 인증 실패 로그에는 비밀번호, JWT, Refresh Token 원문, 내부 예외 상세 또는 기타 민감정보를 포함하지 않는다.

#### 인증·인가 통합 승인 기준

**회원가입**
- 정상 이메일·비밀번호로 회원가입에 성공한다.
- 저장된 비밀번호가 평문과 일치하지 않고 안전한 해시로 저장된다.
- 가입 사용자의 기본 Role이 `USER`다.
- 중복 이메일 가입이 `409 Conflict`로 거부된다.
- 공개 요청으로 `ADMIN` 계정을 생성하거나 Role을 선택할 수 없다.
- 잘못된 입력값 가입이 `400 Bad Request`로 거부된다.

**로그인 및 JWT**
- 올바른 자격 증명으로 Access Token과 Refresh Token이 함께 발급된다.
- 잘못된 비밀번호 및 존재하지 않는 이메일 로그인이 각각 `401 Unauthorized`로 실패한다.
- 정상 Access Token으로 보호 API에 접근할 수 있다.
- 토큰 없이, 변조된 토큰으로, 만료된 토큰으로 보호 API에 접근하면 각각 `401 Unauthorized`가 반환된다.

**Role 인가**
- `USER`가 일반 보호 API에 접근할 수 있다.
- `ADMIN`이 일반 보호 API와 관리자 API에 접근할 수 있다.
- `USER`가 관리자 API에 접근하면 `403 Forbidden`이 반환된다.

**Refresh Token 및 로그아웃**
- 유효한 Refresh Token으로 토큰 재발급에 성공하고 새 Access Token과 새 Refresh Token을 받는다.
- 재발급 후 기존 Refresh Token 재사용이 실패한다.
- 만료·변조된 Refresh Token 재발급이 실패한다.
- 로그아웃 후 폐기된 Refresh Token 재발급이 실패한다.

**보안**
- JWT 비밀키가 소스코드와 Git 저장소에 포함되지 않는다.
- 인증 실패 응답과 로그에 비밀번호, JWT 및 Refresh Token 원문이 노출되지 않는다.

#### 인증 범위 데이터 모델

첫 번째 인증·인가 구현 증분은 아래 두 모델과 관계만 생성한다. 전체 도메인 테이블은 이 증분에서 생성하지 않는다.

- **USERS:** `id`, `email` unique, `password_hash`, `role` (`USER | ADMIN`), `created_at`
- **REFRESH_TOKENS:** `id`, `user_id` FK, `token_hash` unique, `expires_at`, `revoked_at` nullable, `created_at`
- **관계:** `USERS 1:N REFRESH_TOKENS`

**Consequences (testable):**
- 동일 이메일을 가진 두 `USERS` 레코드를 생성할 수 없다.
- 동일 `token_hash`를 가진 두 `REFRESH_TOKENS` 레코드를 생성할 수 없다.
- 모든 `REFRESH_TOKENS.user_id`는 존재하는 `USERS.id`를 참조한다.
- 한 사용자는 여러 Refresh Token 레코드를 가질 수 있고 각 토큰의 만료 및 폐기 상태를 독립적으로 관리할 수 있다.

### 4.2 캐릭터 관리 (Character)

**Functional Requirements:**

#### FR-3: 캐릭터 생성
사용자는 이름·디자인 키워드·성격을 입력해 캐릭터를 생성할 수 있다. 생성 요청 안에서 Spring Boot가 FastAPI `POST /api/ai/commitgotchi`를 동기 호출해 AI 스프라이트 이미지 생성을 시도하며, 실패 시 기본 스프라이트로 대체해 캐릭터 생성 자체는 성공시킨다.

#### FR-5: 캐릭터 수정

**Consequences (testable):**

**Consequences (testable):**

- 학습 점수(점수 변화량)는 지정된 활성 캐릭터에만 반영된다.

### 4.4 AI 일일 레포트 (AI Daily Report)

**Description:** 매일 자정 이후 시스템이 사용자별로 그날 **학습 리포트**와 **그날 이미 채점된 퀴즈 결과 요약**을 모아 AI에 분석을 요청한다. AI는 학습 내용을 분석해 능력치별 점수 변화량(학습 리포트 분석분)을 산출하고, 퀴즈 결과를 종합 코멘트하며, 다음 학습 추천과 추천 퀴즈를 생성한다. **퀴즈 채점 자체는 이 배치가 아니라 제출 즉시(FR-15) 수행된다.** 결과는 매일 오전 9시까지 사용자에게 제공되고 활성 캐릭터에 반영된다. 실현 UJ-1.

**Functional Requirements:**

#### FR-9: 일일 레포트 생성 요청 적재
시스템은 매일 자정 이후 사용자별 AI 일일 레포트 생성 요청을 비동기 큐에 적재한다.

**Consequences (testable):**
- 각 요청은 멱등 식별자(requestId)를 가진다.
- 요청 메시지는 활성 캐릭터 정보, 학습 리포트, 주간 학습 스트릭, 주간 점수 변화량, **그날 이미 채점된 퀴즈 결과 요약(todayQuizResults)** 컨텍스트를 포함한다(스키마는 addendum 참조). 미채점/미제출 답안은 포함하지 않는다.

#### FR-10: AI 일일 레포트 생성
AI는 큐의 요청을 단건 처리해 AI 일일 레포트(분석·점수 변화량·다음 학습 추천·추천 퀴즈)를 생성한다.

**Consequences (testable):**
- 점수 변화량은 능력치 5종(DB·알고리즘·CS·네트워크·프레임워크) 각각에 대해 산출된다.
- 처리 성공 시 결과가 시스템에 전달되고 해당 큐 메시지는 제거된다.
- 처리 실패 시 Fallback 정책(FR-16)이 적용되고 흐름이 중단되지 않는다.

#### FR-11: 점수 변화량의 활성 캐릭터 반영 (일 단위 누적)
시스템은 산출된 점수 변화량을 활성 캐릭터의 능력치에 **일 단위로 누적**한다.

**Consequences (testable):**
- 점수는 일 단위로 누적되며, 주간 점수 변화량은 표시·통계·AI 컨텍스트 용도로만 집계된다(반영 단위 아님).
- 점수 반영과 전투력 재계산은 하나의 일관된 갱신으로 처리되어, 동시성 상황에서도 활성 캐릭터 단일성과 합계 정합성이 깨지지 않는다.
- 반영 후 전투력 = 5개 능력치 총합이 갱신된다.

#### FR-12: 사용자에게 결과 제공
사용자는 매일 오전 9시까지 자신의 AI 일일 레포트(피드백·점수 변화·다음 학습 추천)를 확인할 수 있다.

**Consequences (testable):**
- 결과 미생성/실패 상태도 사용자에게 명확히 표시된다(빈 화면이 아니라 상태 안내).

**Notes:** `[NOTE FOR PM]` **개정:** 퀴즈 채점은 제출 즉시 동기 처리로 변경(FR-15, 흐름 B). 자정 배치(흐름 A)는 학습 리포트 분석·다음 추천·추천 퀴즈 생성만 담당하고, 그날 채점된 퀴즈 결과를 종합 코멘트에만 활용한다(점수 이중계상 금지). 결정 D1·D4 갱신됨(addendum 참조).

### 4.5 퀴즈 (Quiz)

**Description:** AI 일일 레포트가 다음날(오전 9시) 추천 퀴즈를 생성한다. 사용자는 추천 퀴즈를 풀어 답안을 제출하며, **제출 즉시** AI가 모범답안 대비 채점·피드백하고 그 결과가 사용자에게 바로 표시되며 점수가 활성 캐릭터에 즉시 반영된다. 실현 UJ-1.

**Functional Requirements:**

#### FR-13: 추천 퀴즈 제공
사용자는 AI가 생성한 추천 퀴즈를 조회하고 풀 수 있다.

**Consequences (testable):**
- 추천 퀴즈는 전날 학습·답안 분석에 기반해 자정 배치(FR-10)에서 생성되고 오전 9시 노출된다.
- 각 퀴즈는 채점 근거가 될 모범답안을 가진다.

#### FR-14: 퀴즈 답안 제출
사용자는 퀴즈에 답안을 제출한다. 제출 답안은 저장되고 곧바로 채점(FR-15)으로 이어진다.

**Consequences (testable):**
- 제출 시 답안 저장은 동기 처리되며, 이어서 즉시 채점이 트리거된다.
- 미제출 퀴즈와 제출 완료 퀴즈가 구분된다.
- 답안 재제출(수정) 정책이 정의된다(기존 제출 갱신 + 점수 롤백·재반영, 당일 자정 마감). [ASSUMPTION]

#### FR-15: 퀴즈 즉시 채점·피드백 (제출 시 동기)
사용자가 답안을 제출하면 AI가 **즉시** 모범답안 대비 채점하고 피드백을 생성하며, 결과가 곧바로 사용자에게 표시된다.

**Consequences (testable):**
- 서술형 답안에 대해 단순 정답 매칭이 아닌 내용 분석 기반 채점·피드백이 생성된다.
- 채점 결과로 산출된 점수 변화량은 **제출 즉시** 활성 캐릭터에 반영되고 감정·상태 메시지가 갱신된다(흐름 B).
- 퀴즈 점수는 즉시 채점에서 1회만 반영되고, 자정 리포트(FR-10)는 퀴즈 점수를 다시 더하지 않는다(이중계상 금지).
- 채점 실패·지연 시 Fallback 정책(FR-16)이 적용된다 — 답안은 저장되고 "미채점/잠시 후 재시도" 상태로 즉시 응답한다.

### 4.6 AI 처리 안정성 / Fallback

**Description:** AI 처리(이미지 생성·레포트 생성·채점)는 실패할 수 있으므로 각 지점에 기본 처리를 둬 사용자 흐름이 끊기지 않게 한다. 데모에서의 안정성이 핵심 성공 기준이다.

**Functional Requirements:**

#### FR-16: Fallback 처리
AI 처리 실패 시 시스템은 기본 처리로 대체해 흐름을 유지한다.

**Consequences (testable):**
- 스프라이트 이미지 생성 실패 → 기본 스프라이트 세트(동일 3프레임 1x3 레이아웃) 사용, 캐릭터 생성은 성공.
- 일일 레포트 생성 실패 → 기본 처리(점수 변화량 0 또는 기본 레포트)로 대체하고 사용자에게 상태를 안내. [ASSUMPTION: 실패 시 점수 변화량 0]
- 퀴즈 즉시 채점 실패·지연 → 답안 저장 후 미채점 상태로 즉시 응답, 점수 미반영, 사용자에게 "잠시 후 재시도" 안내.
- 어떤 단일 AI 실패도 사용자 흐름(리포트·퀴즈·캐릭터 생성)을 중단시키지 않는다.

### 4.7 랭킹·대시보드 (Ranking & Dashboard)

**Description:** 사용자는 대시보드에서 활성 캐릭터와 학습 현황을 보고, 랭킹에서 전투력 기준 순위를 확인한다. 실현 UJ-3.

**Functional Requirements:**

#### FR-17: 대시보드
사용자는 활성 캐릭터, 능력치 5종, 전투력, 감정·상태 메시지, 최근 학습 현황을 한 화면에서 확인할 수 있다.

**Consequences (testable):**
- 활성 캐릭터가 없을 때도 빈 상태(캐릭터 생성 유도)가 표시된다.

#### FR-18: 랭킹
사용자는 활성 캐릭터 전투력 기준의 랭킹과 자신의 순위를 조회할 수 있다.

**Consequences (testable):**
- 랭킹은 전투력 내림차순으로 정렬된다.

### 4.8 캐릭터 공유 게시판·리뷰 (Sharing)

**Description:** 사용자는 캐릭터를 공유 게시판에 올리고(공유 게시글), 다른 사용자의 공유 캐릭터에 리뷰를 남긴다. 게시글·리뷰는 생성·조회·수정·삭제(CRUD)된다. 실현 UJ-3.

**Functional Requirements:**

#### FR-19: 공유 게시글 CRUD
사용자는 자신의 캐릭터를 공유 게시글로 작성·조회·수정·삭제할 수 있다.

**Consequences (testable):**
- 사용자는 자신의 게시글만 수정·삭제할 수 있다.
- 공유 게시글은 다른 사용자가 조회할 수 있다.

#### FR-20: 리뷰 CRUD
사용자는 공유 게시글에 리뷰를 작성·조회·수정·삭제할 수 있다.

**Consequences (testable):**
- 사용자는 자신의 리뷰만 수정·삭제할 수 있다.

### 4.9 캐릭터 능력치·감정·진화 규칙 (Growth Rules)

**Description:** 학습 결과는 능력치·전투력·감정·진화로 표현된다. 전투력은 능력치 총합이며, 1,000점 도달 시 진화가 한 번 일어난다. 감정은 학습 결과에 따라 변한다.

**Functional Requirements:**

#### FR-21: 능력치·전투력 갱신
시스템은 점수 변화량 반영 후 능력치와 전투력(=총합)을 갱신한다. (FR-11과 연동)

**Consequences (testable):**
- 전투력은 항상 5개 능력치 합과 일치한다.

#### FR-22: 진화
캐릭터의 전투력이 1,000점 이상이 되면 캐릭터가 진화한다.

**Consequences (testable):**
- 진화는 캐릭터당 최대 1회 발생한다.
- 진화 시 캐릭터 스프라이트가 진화형(mature) 프레임 행으로 전환된다(같은 스프라이트시트의 아래 행).
- `[ASSUMPTION]` 진화 시 능력치 보너스 등 규칙: 진화 단계 도달을 시각적 이미지 교체 + 진화 상태 플래그로 표현하고, MVP에서는 추가 능력치 보너스 없이 1,000점 임계 통과 자체를 보상으로 둔다. (검토 후 수정 가능 — §8)

#### FR-23: 감정·상태 메시지 반영
시스템은 학습 결과(점수 변화량/채점 결과)에 따라 캐릭터 감정(기쁨·슬픔·화남)과 상태 메시지를 갱신한다.

**Consequences (testable):**
- 긍정적 학습 결과는 기쁨, 저조하거나 미학습은 슬픔/화남으로 반영된다. [ASSUMPTION: 감정 산정 기준 구체 규칙은 §8]
- 상태 메시지는 활성 캐릭터의 성격(personality)을 반영한다.

## 5. Non-Goals (Explicit)

- commitgotchi는 결제·구독 서비스가 아니다 — v1에 과금/결제 없음.
- 모바일 네이티브 앱이 아니다 — v1은 웹.
- 본격 소셜 네트워크가 아니다 — 친구·팔로우·DM 등 소셜 그래프 없음(공유 게시판·리뷰까지만).
- 실시간 알림/푸시 플랫폼이 아니다 — v1에 푸시/실시간 알림 없음.
- 다국어 학습 플랫폼이 아니다 — v1 콘텐츠·추천은 CS 한국어 중심.
- **첫 번째 인증·인가 구현 증분**에는 이메일 인증, 비밀번호 찾기·재설정, OAuth·소셜 로그인, 다중 인증, Access Token 블랙리스트가 포함되지 않는다.
- **첫 번째 인증·인가 구현 증분**에는 전체 도메인 테이블 생성과 캐릭터·리포트·퀴즈·게시판·Spring AI 기능 구현이 포함되지 않는다.

## 6. MVP Scope

### 6.1 In Scope

- 회원 가입·로그인·JWT 인증·재발급·로그아웃·Role 기반 인가 (FR-1~2, FR-24~28)
- 캐릭터 CRUD + 활성화, 최대 3개·활성 1개 보장 (FR-3~7)
- 일일 학습 리포트 작성·저장 (FR-8)
- AI 일일 레포트 생성(자정 배치): 분석·점수 변화량·다음 학습 추천·추천 퀴즈 (FR-9~13)
- 퀴즈 답안 제출 + **제출 즉시 동기 AI 채점·피드백·점수 반영** (FR-14~15)
- AI 처리 실패 Fallback (FR-16)
- 능력치·전투력·진화·감정 반영 (FR-21~23)
- 전투력 기준 랭킹·대시보드 (FR-17~18)
- 캐릭터 공유 게시글·리뷰 CRUD (FR-19~20)

### 6.2 Out of Scope for MVP

- 결제·구독, 모바일 네이티브 앱, 다국어, 실시간 알림/푸시, 친구·팔로우 등 소셜 그래프.
- CS 외 학습 도메인(어학·자격증) 확장 — 비전 단계.
- 진화 2단계 초과(3단계 이상) — v1은 기본/진화 2단계만.

### 6.3 First Implementation Increment — Spring Boot 인증·인가

전체 MVP 기능 범위는 유지하되, 구현 순서상 가장 먼저 완료할 증분은 Spring Boot 인증·인가다.

**In Scope**
- 인증과 직접 관련된 `users`, `refresh_tokens` 테이블만 생성한다.
- 회원가입, 로그인, JWT Access Token 발급·검증, Refresh Token 발급·Rotation 기반 재발급, 로그아웃을 구현한다.
- `USER`, `ADMIN` Role 기반 인가와 `GET /api/users/me`, `GET /api/admin/ping`을 구현한다.
- FR-1~2 및 FR-24~28의 오류 응답과 통합 승인 기준을 충족한다.

**Out of Scope**
- 이메일 인증, 비밀번호 찾기·재설정, OAuth·소셜 로그인, 다중 인증, Access Token 블랙리스트.
- 전체 도메인 테이블 생성.
- 캐릭터, 리포트, 퀴즈, 게시판, Spring AI 기능 구현.

## 7. Success Metrics

포트폴리오 MVP 기준이므로 데모에서 핵심 가치가 살아 있는지에 맞춘다.

**Primary**

- **SM-1 (AI 품질, 1순위):** 샘플 학습 입력 세트에 대한 채점 정확도·피드백 유용성 정성 평가가 "그럴듯하고 유용하다" 수준. 목표: 평가자 다수가 유용하다고 판단. Validates FR-10, FR-15. [ASSUMPTION: 정성 평가 방식·표본]
- **SM-2 (루프 완결성):** 학습 리포트 작성 → AI 처리 → 캐릭터 성장까지 종단 플로우가 데모에서 끊김 없이 완주. 목표: 데모 시나리오 100% 완주. Validates FR-8~12, FR-21~23.

**Secondary**

- **SM-3 (안정성):** AI 처리 실패를 주입해도 Fallback이 작동해 흐름이 중단되지 않음. 목표: 주입 실패 케이스 전부에서 흐름 유지. Validates FR-16.
- **SM-4 (인증 증분 완결성):** FR-1~2 및 FR-24~28의 인증·인가 통합 승인 기준이 자동화 테스트에서 모두 통과한다. 목표: 명시된 필수 승인 기준 100% 통과.

**Counter-metrics (do not optimize)**

- **SM-C1 (점수 인플레 경계):** 점수 변화량을 후하게 줘 전투력만 빠르게 올리는 방향으로 최적화하지 않는다 — AI 품질(SM-1)을 희생하면서 진화 속도만 높이면 안 됨. SM-1을 상쇄.
- **SM-C2 (게임화 과잉 경계):** 캐릭터 연출·랭킹 몰입을 위해 학습 본질(피드백 정확도)을 흐리지 않는다. SM-2를 상쇄.
