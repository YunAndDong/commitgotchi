# Story BE-2.1: 캐릭터 정규화 스키마와 도메인 모델 도입

Status: done

## Story

As a 백엔드 개발자,
I want 캐릭터와 능력치를 정규화 테이블과 도메인 모델로 관리하고,
so that 후속 캐릭터 생성, 활성화, 성장, 랭킹 기능을 `game_states.state_json`이 아닌 트랜잭션 가능한 System of Record 위에 구현할 수 있다.

## 목적과 범위

- `springboot/` 내부에만 변경을 만든다.
- Flyway `V4__create_characters.sql`로 캐릭터 정규화 테이블과 DB 제약을 추가한다.
- `com.commitgotchi.character.domain` 패키지에 JPA 엔티티, enum, Repository, 도메인 규칙을 추가한다.
- 능력치 5종, 전투력 합계, 진화 플래그, 감정, 이미지 상태, 활성 캐릭터 단일성을 스키마와 도메인 테스트로 고정한다.
- 기존 인증/인가, CORS, Swagger, `/api/game/**` JSON 프로토타입 API는 동작을 유지한다.
- 비범위: 공개 캐릭터 생성/조회/수정/삭제 API, FastAPI 이미지 생성 호출, S3 URL 생성, `GameService`의 projection 전환, `game_states` 제거, Vue/FastAPI 수정.

## Acceptance Criteria

1. **Flyway 캐릭터 스키마 생성**
   - **Given** PostgreSQL 16 Testcontainers 기반 테스트 DB가 비어 있을 때
   - **When** Spring Boot 테스트 컨텍스트가 Flyway를 실행하면
   - **Then** `characters` 테이블이 `V4__create_characters.sql`로 생성된다.
   - **And** `users(id)`를 참조하는 `user_id` FK가 있고 사용자 삭제 시 캐릭터도 삭제된다.
   - **And** `name`, `design_keyword`, `personality`, `emotion`, `status_message`, `is_evolved`, `image_status`, `sprite_sheet_url`, `sprite_meta`, `is_active`, `created_at`, `updated_at`, `version` 컬럼을 가진다.
   - **And** `sprite_meta`는 PostgreSQL `jsonb`로 저장한다.

2. **능력치와 전투력 정합성**
   - **Given** 캐릭터가 생성될 때
   - **When** `stat_db`, `stat_algorithm`, `stat_cs`, `stat_network`, `stat_framework`가 저장되면
   - **Then** 각 능력치는 0 이상의 정수여야 한다.
   - **And** `battle_power`는 항상 다섯 능력치 합과 일치한다.
   - **And** 사용자 직접 수정 대상은 `name`, `design_keyword`, `personality`로 한정되고 능력치, 전투력, 진화 상태, 감정은 시스템 전용 필드로 모델링한다.

3. **상태 enum과 이미지 상태 충돌 흡수**
   - **Given** PRD는 이미지 `PENDING` 흐름을, Architecture는 `READY|FALLBACK` 동기 흐름을 말한다.
   - **When** BE-2.1 스키마와 enum을 설계하면
   - **Then** `CharacterImageStatus`는 `PENDING`, `READY`, `FALLBACK`, `FAILED`를 표현할 수 있어야 한다.
   - **And** BE-2.1은 이미지 상태 전이를 확정하거나 FastAPI를 호출하지 않는다.
   - **And** `CharacterEmotion`은 `JOY`, `SAD`, `ANGRY`만 허용한다.

4. **활성 캐릭터 단일성**
   - **Given** 한 사용자가 여러 캐릭터를 보유할 수 있을 때
   - **When** 둘 이상의 캐릭터가 동시에 `is_active=true`가 되려 하면
   - **Then** PostgreSQL 부분 유니크 인덱스가 사용자당 활성 캐릭터 최대 1개를 강제한다.
   - **And** Repository는 후속 BE-2.2~2.4에서 활성 변경 트랜잭션을 구현할 수 있도록 사용자별 캐릭터 조회와 필요한 경우 비관적 잠금 메서드를 제공한다.

5. **기존 프로토타입 API 보존**
   - **Given** 현재 `GameController`와 `GameService`가 `/api/game/**` 요청을 `game_states.state_json`에 저장하고 있을 때
   - **When** BE-2.1이 완료되면
   - **Then** `/api/game/state`, `/api/game/characters`, `/api/game/characters/{id}/active` 등 기존 경로의 컴파일 및 기존 테스트가 깨지지 않는다.
   - **And** 새 핵심 도메인 규칙을 `game_states.state_json`에 추가하지 않는다.

6. **테스트 완료 기준**
   - **Given** `./gradlew test`가 실행될 때
   - **When** 단위 테스트와 PostgreSQL 통합 테스트가 완료되면
   - **Then** Flyway 버전 `1`, `2`, `3`, `4`가 순서대로 적용된다.
   - **And** 활성 캐릭터 부분 유니크 인덱스, FK cascade, enum/check 제약, 능력치 음수 거부, 전투력 합계 정합성이 검증된다.
   - **And** `LearningCharacter` 도메인 단위 테스트가 기본값, 편집 가능 필드, 시스템 전용 필드 보호, 진화 임계값 규칙을 검증한다.

## Dev Notes

### Developer Context

- 인증 에픽은 완료되어 `users`, `refresh_tokens`, JWT Access Token, Refresh Token Rotation, 로그아웃, Role 인가, CORS, Swagger 프로필 제어가 동작한다.
- 현재 Spring Boot 루트는 `springboot/`이며 `build.gradle`은 Spring Boot `3.3.5`, Java 17, Spring Data JPA, Spring Security, Flyway, PostgreSQL, Testcontainers를 이미 포함한다. 새 라이브러리를 추가하지 않는다.
- 현재 `GameController`는 `/api/game/**` 엔드포인트를 제공하고 `GameService`는 사용자별 `game_states.state_json`에 캐릭터, 리포트, 퀴즈, 게시판 상태를 JSON으로 저장한다. 이 JSON 저장소는 프로토타입 어댑터이며 새 핵심 도메인 규칙을 더하지 않는다.
- `GameService`의 현재 stat key는 프런트 호환을 위해 `algo`, `cs`, `db`, `net`, `fw`다. DB 내부 표준은 PRD/Architecture의 5종 능력치에 맞춰 `stat_algorithm`, `stat_cs`, `stat_db`, `stat_network`, `stat_framework`로 둔다. 후속 projection에서 `algorithm -> algo`, `network -> net`, `framework -> fw` 매핑을 맡는다.
- 마스터 BMad `Story 2.1`은 "첫 캐릭터 생성 및 자동 활성화"지만, Spring Boot 전용 `BE-2.1`은 "캐릭터 정규화 스키마와 도메인 모델 도입"이다. 이 스토리는 Spring Boot 전용 epics를 SSOT로 삼고, 실제 생성 API는 BE-2.2로 넘긴다.
- 이전 PRD는 이미지 생성을 지연 완료되는 `PENDING` 흐름으로 설명하고, Architecture는 동기 HTTP `READY|FALLBACK` 흐름으로 확정했다고 설명한다. BE-2.1은 enum 수용 폭만 열어두고 상태 전이와 FastAPI 계약 구현은 BE-2.5에서 확정한다.
- 기존 공통 오류 응답은 `ErrorCode`와 `GlobalExceptionHandler`가 처리한다. BE-2.1은 공개 API가 없으므로 새 오류 코드를 서두르지 않는다. BE-2.2에서 생성 제한/소유권/검증 API 오류 계약을 추가한다.
- 기존 DB 테스트는 `PostgresIntegrationTest`의 PostgreSQL 16 Testcontainer와 실제 Flyway/JPA validation을 사용한다. 같은 방식으로 검증한다.

### Suggested Schema Contract

```sql
CREATE TABLE characters (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id BIGINT NOT NULL,
    name VARCHAR(40) NOT NULL,
    design_keyword VARCHAR(120) NOT NULL,
    personality VARCHAR(500) NOT NULL,
    stat_db INTEGER NOT NULL DEFAULT 0,
    stat_algorithm INTEGER NOT NULL DEFAULT 0,
    stat_cs INTEGER NOT NULL DEFAULT 0,
    stat_network INTEGER NOT NULL DEFAULT 0,
    stat_framework INTEGER NOT NULL DEFAULT 0,
    battle_power INTEGER NOT NULL DEFAULT 0,
    emotion VARCHAR(20) NOT NULL DEFAULT 'JOY',
    status_message VARCHAR(160) NOT NULL,
    is_evolved BOOLEAN NOT NULL DEFAULT false,
    image_status VARCHAR(20) NOT NULL DEFAULT 'PENDING',
    sprite_sheet_url TEXT NULL,
    sprite_meta JSONB NULL,
    is_active BOOLEAN NOT NULL DEFAULT false,
    version BIGINT NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_characters_user
        FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT ck_characters_name
        CHECK (length(btrim(name)) > 0),
    CONSTRAINT ck_characters_design_keyword
        CHECK (length(btrim(design_keyword)) > 0),
    CONSTRAINT ck_characters_personality
        CHECK (length(btrim(personality)) > 0),
    CONSTRAINT ck_characters_status_message
        CHECK (length(btrim(status_message)) > 0),
    CONSTRAINT ck_characters_stats_non_negative
        CHECK (
            stat_db >= 0
            AND stat_algorithm >= 0
            AND stat_cs >= 0
            AND stat_network >= 0
            AND stat_framework >= 0
        ),
    CONSTRAINT ck_characters_battle_power
        CHECK (battle_power = stat_db + stat_algorithm + stat_cs + stat_network + stat_framework),
    CONSTRAINT ck_characters_emotion
        CHECK (emotion IN ('JOY', 'SAD', 'ANGRY')),
    CONSTRAINT ck_characters_image_status
        CHECK (image_status IN ('PENDING', 'READY', 'FALLBACK', 'FAILED')),
    CONSTRAINT ck_characters_ready_requires_sprite
        CHECK (image_status <> 'READY' OR (sprite_sheet_url IS NOT NULL AND length(btrim(sprite_sheet_url)) > 0))
);

CREATE UNIQUE INDEX uq_one_active_character_per_user
    ON characters (user_id)
    WHERE is_active = true;

CREATE INDEX idx_characters_user_created_at
    ON characters (user_id, created_at DESC);
```

`battle_power` is intentionally a normal column guarded by a CHECK constraint. This keeps JPA mapping simple while still letting PostgreSQL reject inconsistent writes from any code path.

### Architecture Compliance

- API -> application -> domain/repository direction must be preserved. BE-2.1 is mostly `domain` plus migration/tests.
- PostgreSQL is Spring Boot's SoR. FastAPI must not read/write this DB.
- Active character uniqueness belongs at DB level with PostgreSQL partial unique index, not only service logic.
- `ddl-auto=validate` must remain. Do not use Hibernate auto-DDL for migrations.
- `game_states` is a compatibility/prototype store. New normalized character rules belong in `characters`.
- `LearningCharacterRepository` should avoid methods that can accidentally skip ownership checks. Prefer `findByIdAndUserId...` for future API use.

### Library and Framework Requirements

- Use existing project versions from `springboot/build.gradle`: Spring Boot `3.3.5`, Java 17, Spring Data JPA, Flyway, PostgreSQL, Testcontainers.
- Do not upgrade to current Spring Boot/Spring Data lines in this story. Current official docs show newer stable lines, but this project is pinned and should remain dependency-managed by the existing Gradle plugin.
- Spring Data JPA supports `@Lock` on repository query methods and redeclared CRUD methods; use it for future active-character/growth transaction methods where row locking is required.
- PostgreSQL partial unique indexes are the right DB primitive for "only one active character per user"; they enforce uniqueness only for rows matching the predicate.
- Prefer explicit Flyway SQL `CHECK` constraints over Hibernate-generated DDL annotations because the project runs with `ddl-auto=validate`.

### Testing Requirements

- Run `cd springboot && ./gradlew test`.
- Character DB tests must extend `PostgresIntegrationTest`, not H2.
- Assertions should use actual PostgreSQL constraint behavior through `JdbcTemplate` or repositories.
- Existing auth/CORS/Swagger tests are regression guards. Do not narrow their scope to make BE-2.1 pass.
- Avoid test failure messages that print secrets, tokens, Authorization headers, or DB credentials.
