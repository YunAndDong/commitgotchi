# 트러블 슈팅

## 탭마다 커밋고치 위치가 다르게 보임

조사일: 2026-06-14

### 증상

여러 일반 웹페이지 탭에서 같은 활성 커밋고치가 표시되지만, 탭마다 이동 위치와 바라보는 방향이
달라 서로 다른 커밋고치처럼 보였다. 탭을 새로고침하면 해당 탭의 커밋고치만 시작 위치로
돌아갔다.

### 원인

`content-script.js`는 각 탭마다 별도의 실행 환경에서 동작한다. 기존 구현은 각 탭에 커밋고치를
렌더링한 순간 아래 CSS 애니메이션을 독립적으로 시작했다.

```css
animation: commitgotchi-stroll 24s linear infinite;
```

따라서 먼저 열린 탭과 나중에 열린 탭의 애니메이션 시작 시각이 달랐고, 새로고침하거나
백그라운드 탭의 렌더링이 제한될 때도 탭별 진행률이 달라졌다. `chrome.storage.local`에는
`id`, `name`, `emotion`, `isEvolved`만 저장하므로 위치와 방향을 맞출 공통 기준도 없었다.

탭별로 크기가 달라 보일 수 있는 별도 원인은 사이트별 Chrome 확대 비율이다. 커밋고치
스프라이트는 모든 탭에서 `72px`이지만, Chrome은 사이트별 확대 비율을 기억하므로 실제 화면에서
보이는 물리 크기는 달라질 수 있다.

### 적용한 해결

탭마다 시작되는 CSS 이동 애니메이션을 제거하고, 모든 탭이 `Date.now()`를 공통 시각 기준으로
사용해 동일한 24초 주기의 진행률과 방향을 계산하도록 변경했다.

- 같은 시각의 모든 탭은 동일한 이동 진행률과 방향을 사용한다.
- 뷰포트 폭이 다른 탭에서는 동일한 상대 위치를 각 탭 너비에 맞춰 환산한다.
- 새 탭을 열거나 탭을 새로고침해도 현재 공통 진행률에서 바로 시작한다.
- 백그라운드 탭에서 프레임 갱신이 중단되어도 다시 표시되는 첫 프레임에 현재 시각으로 재동기화된다.
- 운영체제의 동작 줄이기 설정에서는 기존처럼 왼쪽 하단에 정적으로 표시한다.

위치 값을 `chrome.storage.local`에 매 프레임 기록하지 않은 이유는 불필요한 저장소 쓰기와 탭 간
이벤트 폭증을 피하기 위해서다. 위치 자체 대신 모든 탭에서 동일한 벽시계를 계산 기준으로
사용하면 저장소 통신 없이도 동기화할 수 있다.

### 검증 방법

1. `npm run build` 후 `chrome://extensions`에서 확장을 새로고침한다.
2. 같은 크기의 일반 HTTP(S) 탭 두 개를 열어 커밋고치 위치와 방향이 일치하는지 확인한다.
3. 한 탭을 새로고침해도 다른 탭과 같은 현재 위치에서 시작하는지 확인한다.
4. 한 탭을 잠시 백그라운드에 둔 후 다시 열어 즉시 같은 위치로 복귀하는지 확인한다.
5. 탭 너비가 다를 때 절대 픽셀 좌표가 아니라 좌우 이동 구간의 상대 진행률이 일치하는지 확인한다.

### 관련 코드

- `vue/public/content-script.js`: 공통 시각 기반 진행률·방향 계산 및 탭별 좌표 환산
- `vue/_tests/extension-storage.mjs`: 독립 CSS 애니메이션 제거와 공통 시각 동기화 계약 검증

## 확장 프로그램 로그인/회원가입: `403 Invalid CORS request`

조사일: 2026-06-14

### 증상

Chrome 확장 프로그램 팝업의 로그인/회원가입 화면에서 다음 오류가 발생했다.

```text
[auth] login/signup failed: Error: Invalid CORS request status=403 code=undefined
```

- 화면 컨텍스트: `index.html#/login?redirect=/`
- 제보된 스택:
  - `assets/index-DoCbvfEH.js:26 (Zd)`
  - `assets/LoginView-Th9NwV8b.js:1 (w)`

### 원인

`403 Invalid CORS request`는 이메일, 비밀번호, 로그인 컨트롤러의 오류가 아니다.
Spring Security의 CORS 필터가 요청의 `Origin`을 허용 목록에서 찾지 못해 인증 로직에 도달하기 전에
거부한 응답이다.

확장 페이지는 다음 형식의 Origin으로 API를 호출한다.

```text
chrome-extension://<extension-id>
```

백엔드는 현재 아래 확장 ID 하나만 신뢰하도록 설정되어 있다.

```text
chrome-extension://daijhhcaecladkkpcjdlfgcokohehhmn
```

기존에는 매니페스트에 `key`가 없어서 압축 해제 확장의 ID가 로드한 폴더의 절대 경로에 따라
달라졌다. 백엔드는 개발자 PC의 `vue/dist` 경로에서 생성된 ID 하나만 신뢰했기 때문에, 다른 PC나
다른 경로에서 로드한 확장은 `403 Invalid CORS request`를 받았다.

`code=undefined`인 이유는 CORS 필터가 애플리케이션 표준 JSON 오류가 아니라
`Invalid CORS request` 평문을 반환하기 때문이다. 프론트엔드의 `client.js`는 평문 응답에서
`code` 값을 추출할 수 없다.

### 이번 조사에서 확인한 상태

- 기존 Chrome 설치 경로는 `/Users/kimyunseok/Desktop/hackaton/commitgotchi/vue/dist`였다.
- 기존 확장 ID와 백엔드 신뢰 ID는 `llnclajenonklpnohgleabfmpaijbgie`였다.
- 현재 실행 중이던 백엔드는 기존 ID의 로그인 preflight를 `200`으로 허용했다.
- 현재 백엔드는 다른 확장 ID의 preflight를 `403 Invalid CORS request`로 거부한다.
- 현재 `vue/dist`의 번들:
  - `assets/index-ClhxywvN.js`
  - `assets/LoginView-XzaUYrJc.js`
- 제보된 오류의 번들 해시는 현재 빌드와 다르다. 즉 오류를 낸 팝업은 현재 `vue/dist`가 아닌
  오래된 확장 번들을 실행하고 있었다.

제보된 오류는 오래된 확장 번들 또는 다른 경로에서 로드되어 ID가 달라진 확장 복사본에서 발생한
것으로 판단했다. 특히 확장 ID가 절대 경로에 의존했던 구조는 다른 개발 환경에서 같은 문제가
반드시 재발할 수 있는 상태였다.

### 적용한 해결

1. `vue/public/manifest.json`에 공개키 `key`를 추가해 확장 ID를 아래 값으로 고정했다.

   ```text
   daijhhcaecladkkpcjdlfgcokohehhmn
   ```

2. Spring Boot의 신뢰 CORS Origin을 새 고정 ID로 변경했다.
3. Vue 테스트에서 매니페스트 공개키로 계산한 확장 ID가 기대값과 일치하는지 검증한다.
4. Spring Boot CORS 단위/통합 테스트도 새 고정 Origin을 검증하도록 변경했다.

### 해결 검증 결과

`vue/dist`를 다시 빌드하고 Spring Boot 컨테이너를 현재 코드로 재빌드·재기동한 뒤 확인했다.

- `vue/dist/manifest.json`의 공개키로 계산한 ID:
  `daijhhcaecladkkpcjdlfgcokohehhmn`
- 새 고정 Origin의 로그인 preflight:
  `200`, `Access-Control-Allow-Origin: chrome-extension://daijhhcaecladkkpcjdlfgcokohehhmn`
- 새 고정 Origin의 실제 로그인 요청:
  CORS를 통과해 애플리케이션 JSON 응답(`VALIDATION_FAILED`)까지 도달
- 이전 경로 기반 Origin의 로그인 preflight:
  `403 Invalid CORS request`
- Vue 전체 테스트: 통과
- Spring Boot CORS 단위/통합 테스트: 통과

### 로컬 적용 절차

고정 ID가 기존 설치 ID와 다르므로 최초 한 번은 기존 Commit-Gotchi 확장을 제거하고 다시 로드해야
한다.

```bash
cd vue
npm run build
```

1. Spring Boot를 현재 코드로 다시 빌드하고 기동한다.
2. `chrome://extensions`에서 기존 Commit-Gotchi 확장을 제거한다.
3. **압축해제된 확장 프로그램을 로드**에서 `vue/dist`를 선택한다.
4. 확장 ID가 `daijhhcaecladkkpcjdlfgcokohehhmn`인지 확인한다.

같은 오류가 계속되면 고정 ID를 넣어 preflight를 검사한다.

```bash
curl -i -X OPTIONS http://localhost:8080/api/auth/login \
  -H 'Origin: chrome-extension://daijhhcaecladkkpcjdlfgcokohehhmn' \
  -H 'Access-Control-Request-Method: POST' \
  -H 'Access-Control-Request-Headers: content-type'
```

정상 응답에는 아래 헤더가 있어야 한다.

```text
HTTP/1.1 200
Access-Control-Allow-Origin: chrome-extension://daijhhcaecladkkpcjdlfgcokohehhmn
Access-Control-Allow-Credentials: true
```

신뢰 ID를 넣었는데도 `403 Invalid CORS request`가 나오면 실행 중인 Spring Boot 이미지가 현재
코드를 반영하지 않은 상태다. 이 경우 백엔드를 다시 빌드하고 기동한다.

```bash
docker compose up -d --build springboot
```

### 재발 방지

- 확장은 항상 `vue/dist`를 압축 해제 확장으로 로드한다.
- Vue 코드를 변경한 뒤에는 `npm run build`와 확장 카드 새로고침을 함께 수행한다.
- `manifest.json`의 `key`를 제거하거나 변경하지 않는다. 변경하면 확장 ID와 CORS Origin이 다시
  달라진다.
- `403 Invalid CORS request`와 `code=undefined`가 함께 보이면 인증 정보보다 실제 요청 Origin과
  백엔드 CORS 허용 목록을 먼저 확인한다.

- `springboot/src/main/java/com/commitgotchi/security/CommitgotchiCorsConfiguration.java`:

- `springboot/src/test/java/com/commitgotchi/security/CorsConfigurationIntegrationTest.java`:
